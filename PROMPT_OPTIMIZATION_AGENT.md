# Задание для ИИ-агента: оптимизированная версия инструмента реестра отклонений ITSM и АС

## Цель

Ты — ИИ-агент. Твоя задача: на основе монолитного Python-скрипта `tema_2025.py` (~7200+ строк) разработать **оптимизированную версию** того же инструмента. Результат должен сохранять полную функциональную совместимость с текущими входными/выходными Excel-файлами и сценариями использования, при этом код должен быть модульным, поддерживаемым и расширяемым.

---

## 1. Назначение инструмента

Инструмент автоматизирует **реестр отклонений по ITSM (IT Service Management) и АС (прикладные системы)** в контексте оценки зрелости и надёжности ИТ-услуг (в т.ч. для дочерних компаний группы). Реализованы **три режима работы**:

| Режим | Название | Кратко |
|-------|----------|--------|
| **0** | Создание анкет ИТ-услуг | Генерация отдельных Excel-анкет по каждому прикладному сервису из реестра объектов. |
| **1** | Генерация реестра | Сводный отчёт по опроснику ITSM и (при наличии) анкетам АС: реестр отклонений, индексы надёжности, показатели. |
| **2** | Ежеквартальный инкремент после ЕОИТ в ДЗО | Расчёт выполненного/запланированного по кварталам на основе Единой дорожной карты (ЕДК/ЕРО), опросника и базы знаний; формирование отчёта ЕРО. |

---

## 2. Требования к оптимизированной версии

### 2.1 Архитектура и структура кода

- **Модульность**: разбить логику на пакет модулей (например: `cli`, `modes`, `io`, `dates`, `reports`, `config`). Один главный точка входа (CLI или `main`).
- **Режимы как отдельные модули или чёткие подмодули**: каждый режим (0, 1, 2) — отдельный обработчик с явными входами/выходами, без дублирования больших блоков кода между режимами.
- **Общие компоненты вынести в общую библиотеку**: работа с датами/кварталами, загрузка/сохранение Excel, справочники (ИНН, ITSM-коды, критичность), валидация опросника и ЕДК.
- **Конфигурация**: пути к папкам (`no_hand`, `AS`), имена файлов (Опросник, База_знаний, ЕДК, ЕРО), ожидаемая версия опросника (6.1.), список листов и столбцов — вынести в конфиг (например YAML/JSON или dataclass/константы в одном месте). Должна быть возможность переопределения путей/имен без правки кода.
- **Типизация**: использовать type hints для аргументов и возвращаемых значений ключевых функций; для сложных структур — TypedDict или dataclass где уместно.
- **Логирование**: заменить часть `print` на модуль `logging` с уровнями (INFO для прогресса, WARNING/ERROR для ошибок). Вывод в консоль оставить читаемым для пользователя.
- **Обработка ошибок**: явная обработка отсутствующих файлов/листов, некорректных данных; понятные сообщения об ошибках и при необходимости коды выхода (например через `sys.exit(1)`).

### 2.2 Интерфейс

- **CLI**: один главный скрипт (например `run.py` или `tema.py`) с подкомандами или флагом режима, например:
  - `python -m tema create-forms` (режим 0),
  - `python -m tema build-registry` (режим 1),
  - `python -m tema quarterly-increment` (режим 2).
- **Ввод параметров**: где возможно — аргументы CLI и опции (например `--company`, `--block`, `--config`). Интерактивный ввод (input) оставить только там, где это действительно нужно (например подтверждение перезаписи файла).
- **Перезапись файлов**: при существующем выходном файле — запрос подтверждения (y/n); при запуске в неинтерактивном режиме — флаг `--overwrite` или `--no-overwrite` (по умолчанию не перезаписывать).

### 2.3 Качество и тесты

- **Тесты**: добавить unit-тесты для:
  - функций работы с датами/кварталами (`get_current_quarter`, `date_to_quarter_year`, `find_future_quarters`, `add_quarters`, `find_min_quarter`);
  - маппинга критичности и кодов (ITSM, A/B/C → Критичный/Высокий/Умеренный);
  - формирования имён файлов и путей.
- **Документация**: README с описанием режимов, структуры входных данных, примеров запуска и требований к окружению (Python 3.x, зависимости из requirements.txt).

### 2.4 Совместимость (обязательно)

- **Форматы Excel**: выходные книги должны открываться в текущих процессах без изменений: те же имена листов, те же столбцы и структура таблиц, те же формулы/стили, где они критичны для потребителей отчёта. Входные файлы — те же листы и ожидаемые заголовки (см. раздел 3).
- **Имена и пути выходных файлов**:
  - Режим 0: `AS/Анкета - ИТ-услуга-<Название_сервиса>.xlsx`.
  - Режим 1: `Расчет_<квартал>_<имя_компании>.xlsx` (имя компании без суффикса «ЕРО» и без расширения в пути при сохранении — имеется в виду тот же файл с расширением .xlsx).
  - Режим 2: `<Компания>_<OP|BO|BC и выше>_ЕРО.xlsx`.
- **Кодировка**: UTF-8 для исходников и по умолчанию для чтения/записи текста в файлах.

## 3. Спецификация данных (входы/выходы по режимам)

### 3.1 Общие входные сущности

- **Рабочая директория**: скрипт предполагает запуск из каталога, где лежат файлы «Опросник», «База_знаний», при режиме 2 — «ЕДК»/«ЕРО»; подкаталоги: `no_hand`, `AS`.
- **Опросник** (xlsx): лист «Надёжность»:
  - Строка 1: название опросника, версия (формат X.X.X.X.X), ИНН.
  - Строка 3: максимальная критичность ИТ-сервиса (OP / BO / BC и выше / Нет подобных ИТ-сервисов), ФИО ответственного.
  - Данные с 4-й строки: ID вопроса, уровень готовности в зависимости от критичности АС, уровень отклонения, ответ с учётом артефакта (авто). Для сопоставления с ЕДК к кодам ITSM может добавляться префикс DTN-.
- **База_знаний** (xlsx):
  - Лист «База»: Код вопроса, Выявленное отклонение, Негативные последствия, Рекомендации, Артефакт.
  - Лист «База АС»: Номер вопроса, Ответ, Риск, Рекомендация (и при необходимости коды для уровней отклонения).
- **Справочник компаний (ИНН → блок, название)** должен быть включён в код или конфиг. Формат: словарь `dict_inn_company`: ключ — ИНН (строка), значение — список `[блок, название_компании]`. Примеры записей: `"7736632467": ["Технологии", "Сбербанк-Технологии"]`, `"7707308480": ["КИБ", "Сбер А"]` — полный список взять из исходного скрипта (dict_inn_company).
- **Критичность**: OP, BO, BC и выше; в шаблонах анкет — маппинг с английских обозначений (Office Productivity → OP, Business Operational → BO, Business Critical → BC, Mission Critical → MC).
- **ITSM-процессы**: фиксированный набор кодов (ITSM-INC, ITSM-CATM, ITSM-CFG, ITSM-SLM, ITSM-PRB, ITSM-ERM, ITSM-RFF, ITSM-ICHG, ITSM-UPD, ITSM-AVL, ITSM-M&E, ITSM-CONT, ITSM-CAP, ITSM-SD, ITSM-MNTW) и их русские названия; префикс DTN- для сопоставления с ЕДК. Коды 5492.* для маппинга с опросником — при необходимости сохранить из исходного кода.

### 3.2 Режим 0 — Создание анкет ИТ-услуг

- **Входы**:
  - Папка `no_hand`: файл, в имени которого есть «Анкета_объектов» (например «Анкета_объектов_оценки.xlsx»), лист «Реестр объектов обследования». Столбец с заголовком «Код» задаёт начало таблицы; нужны колонки: Код, Тип объекта, Название, Критичность сервиса, Описание.
  - Шаблоны в `no_hand`: «Анкета - ИТ-услуга.xlsx», «Анкета - ИТ-для скрипта.xlsx» (листы «Технологии», «Компоненты ИТ-услуги», «Справочники» для валидаций).
  - Ввод пользователя: название компании, блок, эксперт по надёжности (ДТН), куратор по архитектуре (ДКА).
- **Логика**:
  - Отобрать только строки с типом объекта «Прикладной сервис».
  - Для каждого такого сервиса: открыть копию шаблона «Анкета - ИТ-услуга.xlsx», подставить в лист «Характеристики ИТ-услуги» компанию, год, блок, экспертов, название сервиса, код, критичность (OP/BO/BC/MC), описание; скопировать валидации из «Анкета - ИТ-для скрипта.xlsx» для листов «Технологии» и «Компоненты ИТ-услуги»; сохранить в `AS/Анкета - ИТ-услуга-<очищенное_название>.xlsx`.
- **Выход**: файлы в папке `AS`. Имена без символов `\ / : * ? " < > |`.

### 3.3 Режим 1 — Генерация реестра (ITSM + АС или только ITSM)

- **Входы**:
  - Файлы «Опросник», «База_знаний» в текущем каталоге.
  - Опционально: папка `AS` с заполненными анкетами (xlsx) — листы «Титульный», «Технологии» (ячейки C4, C5, D21, F21; столбцы A, C, F, P, N, H и т.д. по логике исходного скрипта).
- **Логика**:
  - Проверить версию опросника (ожидается 6.1.); проверить указанную критичность; определить имя компании по ИНН из справочника.
  - Загрузить «База» и «Надёжность» (с 4-й строки); при необходимости добавить префикс DTN- к кодам ITSM; сформировать реестр отклонений ITSM из базы знаний и ответов «Нет».
  - Если есть папка `AS`: прочитать все xlsx, извлечь отклонения по технологиям, сопоставить с «База АС», подсчитать по уровням Критичный/Высокий/Умеренный; посчитать суммы требований по уровням надёжности 3 и 4 для индекса.
  - Рассчитать индексы надёжности и готовности по процессам ITSM и по АС; сформировать листы отчёта с реестрами и сводками.
- **Выход**: один файл `Расчет_<текущий_квартал>_<имя_компании>.xlsx` (имя компании — без суффикса «ЕРО»). При существующем файле — запрос перезаписи (или флаг в неинтерактивном режиме).

### 3.4 Режим 2 — Ежеквартальный инкремент после ЕОИТ в ДЗО

- **Входы**:
  - «Опросник» (версия 6.1.), файл «ЕДК» или «ЕРО» (лист «Единая дорожная карта»), «База_знаний»; при необходимости папка `AS`.
- **Логика**:
  - Загрузить ЕДК: колонки Код объекта, Код вопроса ЕОИТ, Задача/отклонение, Вхождение в уровни готовности, Рекомендованные мероприятия, Уровень принятия решения, Уровень критичности, Запланированное мероприятие, даты начала/окончания, Ответственный, Статус, Комментарии и т.д. Для кодов ITSM в «Код вопроса ЕОИТ» добавить префикс DTN-.
  - Отбирать объекты вида «АС» по кодам из «База АС». Даты приводить к формату квартала (например 1Q2025) с помощью общей функции парсинга дат.
  - Подсчитать общее число отклонений по критичности и число «выполненных» (статусы «(ПАО) Устранено - подтверждено ПАО», «(ПАО) Исключение из реестра - подтверждено ПАО»). Определить будущие кварталы по датам окончания незакрытых отклонений.
  - Сопоставить ЕДК с опросником; по кварталам рассчитать выполнено/всего по процессам ITSM и по АС; рассчитать индекс надёжности и уровни готовности 3 и 4 по кварталам, включая прогноз по будущим кварталам.
  - Заполнить книгу с листами: реестр отклонений из ЕДК, сводки по процессам, по АС, квартальный инкремент, прогноз.
- **Выход**: файл `<Компания>_<OP|BO|BC и выше>_ЕРО.xlsx`. При существующем файле — запрос перезаписи (или флаг).

## 4. Работа с датами и кварталами

Реализовать и протестировать единый модуль (например `dates` или `quarters`):

- **Текущий квартал**: формат `nQYYYY` (например 2Q2025).
- **Парсинг**: строки вида `1кв.2025`, `Q1 25`, `июн.2025`, `31.03.2025`, `2025-07-31` и т.п. → квартал в формате `nQYYYY`. Поддержать русские и английские сокращения месяцев (словарь MONTHS).
- **Функции**: `get_current_quarter()`, `extract_quarter_year(q_str)`, `find_future_quarters(current_q, all_quarters)`, `date_to_quarter_year(date_str)`, `add_quarters(initial_date, quarters_to_add)`, `find_min_quarter(list_of_quarter_strings)`.
- Поведение должно совпадать с исходным скриптом (в т.ч. обработка некорректных дат и пустых значений).

## 5. Зависимости

Сохранить совместимость с используемыми библиотеками:

- `openpyxl` (чтение/запись Excel, стили, валидации)
- `sheet2dict` (Worksheet, xlsx_to_dict с select_sheet, skiprows, data_only)
- `python-pptx` (если сохраняется генерация pptx в режимах 1 и 2)
- `pandas` (если используется в текущей логике)
- `dateutil.parser` для парсинга дат
- Остальные: `requests`, `PIL` — только если реально используются в актуальных режимах; иначе вынести в опциональные зависимости или удалить из основного пути.

В оптимизированной версии зафиксировать зависимости в `requirements.txt` с указанием версий по возможности.

## 6. Критерии приёмки

1. Все три режима (0, 1, 2) выполняются и производят выходные файлы с ожидаемыми именами и структурой листов.
2. Результаты открываются в тех же процессах, что и выходы текущего скрипта (ручная проверка или регрессионные тесты на фиксированных примерах).
3. Код разбит на модули; общая логика (даты, справочники, чтение/запись Excel) не дублируется между режимами.
4. Конфигурация (пути, имена файлов, версия опросника) вынесена и при необходимости переопределяется без правки кода.
5. Реализован CLI с подкомандами или флагами для режимов 0, 1, 2; интерактивный ввод сведён к необходимому минимуму.
6. Есть unit-тесты для функций дат/кварталов и по возможности для маппингов и формирования путей.
7. README описывает установку, запуск и форматы входных данных.
